# 硬件接口契约

## 目标

把分散的硬件证据整理成一份简洁、可审查、可执行的接口契约，让固件工程师能实现，让评审者能追溯。宁可明确写出不确定性，也不要编造确定性。

主要输出为 `hardware-interface-contract.json`；需要时再附带一份 Markdown 摘要。

## 流程

1. 盘点证据：列出用到的每份手册、原理图页、引脚表、固件文件、问题单、日志或用户说明。
2. 抽取接口：识别芯片、连接器、网络、总线、地址、引脚、复位、时钟、中断、启动配置、电源轨、时序规则和特殊电气约束。
3. 交叉校验：把手册要求、原理图连接和当前固件假设逐项对照，标出冲突、缺失证据和高风险推断。
4. 编写契约：按 `references/contract-format.md` 的字段和不确定性表达方式输出。
5. 校验结构：只要产出 JSON，就运行 `scripts/validate_contract.py <contract.json>`。
6. 汇报可行动缺口：列出需要硬件、固件或实验室确认的事项，并尽量给出测量点或源码位置。

## 证据规则

- 关键结论都要带短证据引用，例如 `datasheet p.34`、`schematic J3`、`board.c:112` 或 `user note`。
- 未验证假设使用 `confidence: "low"`，并在 `evidence` 中说明原因。
- 不要悄悄化解冲突；把冲突写入 `open_questions` 或 `risks`。
- 尽量保留原始信号名；规范化名称只能作为别名补充。
- 如果缺少资料，先输出部分契约，并明确列出缺失来源。

## 契约内容

有证据时包含这些区域：

- `components`：器件型号、角色、封装、电压域、关键手册来源。
- `interfaces`：总线类型、主从角色、引脚、地址、片选、速率、模式、上拉和端接。
- `pins`：主控引脚、网络名、方向、复用功能、复位状态、电气限制。
- `power`：电源轨、上电顺序、使能脚、电流说明、欠压或爬升约束。
- `boot_reset`：复位脚、启动配置、必要延迟、振荡器或时钟依赖。
- `registers`：只记录初始化、可靠性或板级差异所必需的寄存器。
- `firmware_tasks`：驱动修改、设备树/头文件更新、测试、日志和工厂检查。
- `risks`：矛盾、未验证假设、裕量风险和可能的 bring-up 失败点。
- `open_questions`：需要硬件、版图、供应商资料或测量回答的具体问题。

## 输出风格

保持直接、工程化，不写宽泛教程。优先放实现最相关的信息：

1. 执行摘要：3 到 6 条。
2. 契约 JSON 路径和校验结果。
3. 冲突和开放问题。
4. 建议的固件或实验室下一步。

## 资源

- 起草或评审 JSON 结构时使用 `references/contract-format.md`。
- 检查必需顶层字段和常见结构错误时运行 `scripts/validate_contract.py`。
